# AGENTS.md

## О проекте

Это статический фронтенд-проект без сборщика и без серверной части в репозитории.

Текущий стек:
- HTML5: набор отдельных страниц в корне проекта (`index.html`, `home.html`, `projects.html`, `profile.html` и др.).
- CSS3: единый файл стилей [`styles.css`](/Users/egorneustroev/Documents/piris/styles.css).
- Vanilla JavaScript: единый файл логики [`script.js`](/Users/egorneustroev/Documents/piris/script.js).
- Assets: изображения, SVG, PDF и превью лежат в директории [`assets`](/Users/egorneustroev/Documents/piris/assets).
- Внешние зависимости на клиенте: Google Fonts, шрифт `Manrope`.

Признаки текущей архитектуры:
- Нет `package.json`, сборки, линтера и тестового контура.
- Страницы связаны обычными ссылками между `.html` файлами.
- Версионирование CSS/JS подключений делается query-параметром вида `?v=...`.
- Основной UI строится на общих CSS-классах и `data-*` атрибутах для JS-поведения.

## Структура

- HTML-страницы лежат в корне проекта.
- Общие стили сосредоточены в [`styles.css`](/Users/egorneustroev/Documents/piris/styles.css).
- Общие интеракции сосредоточены в [`script.js`](/Users/egorneustroev/Documents/piris/script.js).
- Медиа, иконки, фотографии, PDF и превью хранятся в [`assets`](/Users/egorneustroev/Documents/piris/assets).

## Правила кода

### HTML

- Использовать семантическую разметку: `main`, `section`, `header`, `nav`, `aside`, `form`, `button`.
- Сохранять существующий стиль форматирования: 2 пробела на уровень вложенности.
- Новые страницы делать в том же формате, что и существующие: общий `head`, подключение `Manrope`, общий [`styles.css`](/Users/egorneustroev/Documents/piris/styles.css), общий [`script.js`](/Users/egorneustroev/Documents/piris/script.js).
- Для интерактивных элементов использовать `data-*` атрибуты как основной контракт между HTML и JS.
- Для декоративных иконок оставлять `alt=""` и `aria-hidden="true"`, как уже сделано в навигации.
- Пользовательский текст и интерфейсные подписи писать на русском, если нет явной причины делать иначе.

### CSS

- Все глобальные токены хранить в `:root` и переиспользовать существующие CSS-переменные перед добавлением новых.
- Поддерживать текущую визуальную систему: светлая палитра PIRIS, мягкие градиенты, скругления, тени, акцентный зелёный бренд-цвет.
- Новые классы именовать в текущем стиле: понятные составные имена через дефис (`sidebar-toggle-button`, `profile-dropdown-link`).
- Не вводить инлайновые стили, если задачу можно решить через общий CSS.
- При расширении компонентов сначала проверять, можно ли встроиться в существующие блоки, а не создавать дублирующий стиль.

### JavaScript

- Использовать только vanilla JS без внешних библиотек, пока в проекте не появится отдельное решение об изменении стека.
- Новую логику добавлять в [`script.js`](/Users/egorneustroev/Documents/piris/script.js) в том же стиле: ранние `return`, понятные имена переменных, работа через `querySelector`/`querySelectorAll` и `data-*`.
- Перед навешиванием поведения проверять наличие DOM-элемента, если код не гарантированно запускается на всех страницах.
- Не завязывать код на хрупкие текстовые селекторы; использовать классы и `data-*`.
- Если логика относится к конкретному UI-блоку, держать рядом связанные константы, функции и обработчики.
- Сохранять текущий стиль данных и контента в JS: объектные конфиги, массивы правил, локализованное форматирование через `Intl.NumberFormat("ru-RU")`.

## Практика изменений

- Перед добавлением новых файлов сначала проверить, можно ли расширить существующие [`styles.css`](/Users/egorneustroev/Documents/piris/styles.css) и [`script.js`](/Users/egorneustroev/Documents/piris/script.js).
- При изменении CSS/JS не забывать обновлять query-версию в HTML-подключениях, если нужно явно пробить кэш.
- Новые ассеты складывать в логичные подпапки внутри [`assets`](/Users/egorneustroev/Documents/piris/assets) по аналогии с текущей структурой.
- Если добавляется новая страница, она должна быть связана с остальной навигацией проекта обычными ссылками.
- Поскольку тестов и линтера нет, после изменений обязательно вручную проверять ключевые сценарии в браузере.

## Что считать нормой для этого репозитория

- Простой входной порог: проект должен открываться как набор статических страниц.
- Минимум магии: без скрытой сборки, без неочевидных зависимостей, без фреймворк-специфичных решений.
- Максимум переиспользования: один набор базовых стилей, один JS-файл, единые паттерны интерфейса.
