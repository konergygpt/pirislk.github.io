# Контекст прототипа PIRIS

У меня есть HTML-прототип личного кабинета PIRIS.

PIRIS - это платформа для проектировщиков, которые закладывают типовые электротехнические решения PIRIS в свои проекты. Прототип сейчас сделан как статический многостраничный фронтенд без сборщика: обычные HTML-страницы, один общий `styles.css`, один общий `script.js`, папка `assets` с иконками, изображениями и PDF-документацией.

Целевая аудитория личного кабинета - проектировщики. Основной сценарий: проектировщик регистрируется, заходит в кабинет, подбирает типовые решения PIRIS, добавляет их в проект, отправляет проект на проверку, отслеживает статус и видит потенциальное или подтвержденное вознаграждение.

## Текущий стек

- Статический HTML/CSS/vanilla JS.
- Без React, без backend, без сборщика.
- Все страницы лежат отдельными `.html` файлами.
- Общие стили в `styles.css`.
- Общая логика в `script.js`.
- Дизайн ориентирован на desktop, мобильную адаптацию решили пока не делать.

## Основные страницы прототипа

- `index.html` - вход в личный кабинет.
- `register.html` - регистрация.
- `forgot-password.html` - восстановление пароля.
- `home.html` - главная страница кабинета.
- `projects.html` - список проектов.
- `project-create.html` - создание проекта.
- `project-submit.html` - отправка проекта и добавление решений.
- `project-detail.html` - карточка проекта.
- `project-company.html` - данные компании или проекта.
- `solutions-catalog.html` - каталог типовых решений PIRIS.
- `rewards.html` - статистика, выплаты, проекты в расчете.
- `grades.html` - уровни и грейды проектировщика.
- `requisites.html` - реквизиты для выплат.
- `profile.html` - профиль пользователя.
- `faq.html` - FAQ для проектировщиков.

## Навигация

- Внутри кабинета есть левое меню: Главная, Проекты, Каталог решений, Статистика, FAQ.
- В верхней панели есть уведомления, профиль пользователя и выпадающее меню профиля.
- Проектировщик может переходить между разделами обычными ссылками.

## Визуальный стиль

- Светлая интерфейсная палитра.
- Основной брендовый цвет - зеленый/бирюзовый.
- Шрифт Manrope.
- Интерфейс похож на рабочий SaaS-кабинет: панели, таблицы, статусы, фильтры, карточки, списки.
- Нужно уходить от гигантских отступов и перегруженных карточек.
- Нужен более строгий, рабочий, понятный интерфейс для проектировщика, а не маркетинговая лендинговая подача.

## Важное дизайн-решение

- Мобильная версия пока не нужна.
- Все проектировщики предполагаемо работают с desktop/laptop.
- Интерфейс можно проектировать desktop-first.
- Приоритет: читаемость, плотность, быстрый скан, понятные статусы и действия.

## Страница `projects.html`

- Сейчас список проектов сделан в виде таблицы/структурированного списка.
- Есть поиск по проектам.
- Есть фильтры: Все, Активные, На проверке, Требуют действий.
- В таблице есть колонки: ID, Проект, Статус, Дата создания, Активность.
- Строки проектов кликабельные и ведут в карточку проекта.
- Примеры статусов: На проверке, Требует уточнения, Подтвержден, Черновик.
- Этот формат таблицы нравится больше, чем большие карточки.

## Страница `rewards.html`

- Это раздел статистики и выплат.
- Сверху есть фильтры: Период, Тип решений, Этап проекта.
- Есть графики и виджеты:
  - Динамика выплат.
  - Статусы проектов.
  - Типы решений: ВРУ, АВР, ГРЩ, ШУН.
- Ниже список проектов в расчете. Его недавно пересобрали из карточек в табличный формат по аналогии с `projects.html`.
- Таблица выплат сейчас содержит колонки: Проект, Статус, Решения PIRIS, Этап, Сумма.
- Из старых карточек убрали нижнюю зону "Следующий шаг / Открыть проект", потому что она перегружала визуал.
- Нужно дальше думать, как сделать статистику полезной, не превращая страницу в перегруженный дашборд.

## Страница `faq.html`

- FAQ предназначен только для проектировщиков.
- Цель FAQ: снизить нагрузку на поддержку, объяснить процесс, помочь с выплатами, каталогом и регистрацией проектов.
- Не нужны аккордеоны с "+", где нужно раскрывать ответ.
- Формат: сразу вопрос и короткий ответ.
- Вопросы разбиты на смысловые блоки:
  - Начало работы.
  - Проекты и решения.
  - Проверка и выплаты.
- У блоков есть маленькие иконки слева.
- Ответы должны быть короткими, простыми, человеческими.
- Важно не превращать FAQ в базу знаний. Подробные инструкции, скриншоты "правильно/неправильно" и длинные кейсы лучше позже вынести в отдельный раздел гайдов или базы знаний.

## Бизнес-логика пока не до конца определена

- Точные правила выплат могут еще меняться.
- Точные критерии проверки проектов могут еще меняться.
- Поэтому в интерфейсе и текстах лучше избегать слишком жестких обещаний.
- Формулировки лучше делать осторожными: "после проверки", "если потребуется уточнение", "статус можно посмотреть в карточке проекта".

## Основная логика продукта

1. Проектировщик регистрируется.
2. Заполняет профиль и при необходимости реквизиты.
3. Заходит в каталог типовых решений PIRIS.
4. Выбирает решения, которые хочет заложить в проект.
5. Регистрирует проект и добавляет решения.
6. Отправляет проект на проверку.
7. Команда PIRIS проверяет проект.
8. Проект получает статус:
   - Черновик.
   - На проверке.
   - Требует уточнения.
   - Подтверждено.
9. Если все подтверждено, проект попадает в расчет вознаграждения или реестр выплат.
10. Проектировщик видит сумму и статус выплаты в статистике.

## Типовые решения PIRIS

- ВРУ.
- АВР.
- ГРЩ.
- ШУН.
- В каталоге решений есть карточки решений, изображения, документация, PDF, чертежи, схемы и спецификации.

## Задача для брейншторма

Нужно побрейнштормить доработки этого прототипа: UX, структуру страниц, тексты, сценарии проектировщика, статусы, выплаты, FAQ/базу знаний и общую продуктовую логику.

Лучше брейнштормить не "как сделать красивее", а по рабочим сценариям:

- Где проектировщик теряется?
- Где поддержка будет получать вопросы?
- Где статус непонятен?
- Где слишком много текста?
- Где не хватает действия?
